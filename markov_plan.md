# План Модернизации Марковского Ядра: Универсальный Контекстный Движок (Source of Truth)

> Центральный рабочий документ и инженерный план для агентов по переходу к Универсальному Вероятностному Марковскому Ядру.
> Роль: детальное руководство для безопасной, бесшовной интеграции процедурного текстового движка без хардкода, без костылей и без внешних runtime-зависимостей.

---

## 1. Архитектурная Философия и Контракт

### 1.1. Единый Source of Truth для процедурной речи
Все нескриптованные (не написанные вручную сценаристом для сюжетных квестов) текстовые поверхности игры обслуживаются **одним универсальным движком**:
1. **Диалоги с NPC** (`src/systems/dialogue.ts`)
2. **Баблы и Ambient Barks в мире** (`src/systems/ai/barks.ts`)
3. **Стеносводка / Логи речи** (`src/systems/markov_log_speech.ts`, `src/systems/world_log.ts`)
4. **Инфосеть Демос: посты и реакции** (`src/systems/demos_posts.ts`, `src/systems/demos.ts`)
5. **Процедурные слухи и квесты** (`src/systems/rumor.ts`, `src/systems/quests.ts`)
6. **[НОВОЕ] Процедурные записки, дневники и документы** (`document_flavor`, `lore_note`)

### 1.2. Разделение Offline (Обучение) и Runtime (Браузер)
**Железное правило:** В браузере (`dist/assets/index.js`) нет доступа к файловой системе `node:fs` и нет времени на парсинг сырых текстов.
- **Offline / Build-Time (Обучение):** Литературные корпуса (`src/data/training_corpus/*.txt`: Стругацкие, Самосбор, Метро, S.T.A.L.K.E.R., советские инструкции ГО) используются **исключительно офлайн-скриптами** (`scripts/`) для дистрибутивного анализа, кластеризации и майнинга синтаксических скелетов.
- **Runtime / Browser (Исполнение):** В итоговый бандл игры (`dist/`) сырые `.txt` **НЕ попадают**. Вместо них в код (`src/data/markov_text.ts` / `src/data/markov_compiled_matrix.ts`) вшивается предкомпилированная, ультракомпактная, типизированная Марковская матрица (N-граммы, санитизированные категории с PCA-весами и синтаксические скелеты).
- В рантайме генератор работает с **нулевыми I/O аллокациями**, мгновенно вычисляя фразу по `SeedRng`.

### 1.3. Строгая Каноничность и Санитизация
Внешние литературные произведения дают только **грамматический ритм и синтаксические скелеты** (`<СУБЪЕКТ> -> <МЕСТО> -> <УГРОЗА>`).
Все чужеродные имена собственных и франшиз (`ЧАЭС`, `Сидорович`, `Контролер`, `Снорк`, `ВДНХ`, `Полис`, `Меченый`, `Ганза`) принудительно отсекаются или проецируются на абстрактные категории при офлайн-компиляции. В сгенерированный текст попадают **только аутентичные сущности лора ГИГАХРУЩА и САМОСБОРА**: `проходчик`, `дежурный электрик`, `ликвидатор`, `лаборант НИИ`, `бетонник`, `черная слизь`, `процедурная аномалия`, `гермодверь`, `вентиляционная шахта`, `дозиметр ДП-5В`, `чистый фильтр АФУ-300`, `пайковый концентрат`, `гаусс-пушка`, `малахитовый артефакт`.

## 2. Математическая Модель: Изоморфизм и Векторные Веса (PCA)

Вместо жестких шаблонов «если X, скажи Y» используется вероятностный выбор атомов по расстоянию между вектором атома и вектором игрового контекста (`MarkovTextContext`).

### 2.1. Категориальная проекция (Theory of Categories)
Атомы в словаре разделены не просто на части речи, а на функциональные категории Гигахруща:
- `SUBJ` (`<СУБЪЕКТ>`): профессии и роли (`ликвидатор`, `слесарь-параноик`, `домком`, `мясник`, `обыватель`).
- `PLACE` (`<МЕСТО>`): локации сектора (`гермодверь`, `вентиляционная шахта`, `трансформаторная ячейка`, `продувочная прачечная`).
- `THREAT` (`<УГРОЗА>`): опасности бетона (`Самосбор`, `черная слизь`, `бетонник`, `пси-излучение`, `удушливый газ`).
- `ITEM` (`<ЦЕННЫЙ_ПРЕДМЕТ>`): хабар и ресурсы (`дозиметр ДП-5В`, `чистый фильтр АФУ-300`, `пайковый концентрат`, `гаусс-пушка`).
- `FACTION` (`<ФРАКЦИЯ>`): организации (`Служба ликвидации`, `Гильдия слесарей`, `НИИ "Щит"`).
- `ACTION` (`<ДЕЙСТВИЕ>`): глаголы выживания и ремонта (`смазывали поворотные механизмы`, `задраили шлюз`, `сдали карточки`).

### 2.2. Векторный скоринг в рантайме (`atomContextScore`)
Каждый атом хранит компактный вектор PCA-осей (числа от -1.0 до 1.0):
1. **Ось Опасности (`pcaDanger`)**: `-1.0` (тишина/быт) <---> `+1.0` (смертельная угроза/Самосбор/монстр).
2. **Ось Ценности (`pcaWealth`)**: `-1.0` (мусор/пустые карманы) <---> `+1.0` (редкий хабар/артефакты/гаусс).
3. **Ось Истощения (`pcaNeed`)**: `-1.0` (сытость/норма) <---> `+1.0` (голодный обморок/жажда/травма).

При генерации слота в `src/systems/markov_text.ts` вероятность выбора атома $A$ для контекста $C$ вычисляется по формуле:
$$Score(A, C) = W_{base} + \alpha \cdot \text{bitCount}(Tags_A \cap Tags_C) + \beta \cdot \text{CosineSim}(V_{PCA}(A), V_{PCA}(C))$$
где $V_{PCA}(C)$ формируется на лету из `context.dangerBand`, `context.wealthBand`, `context.needBand`, `context.recentTrauma` и `context.isSamosborActive`.

## 3. Рабочие Пакеты для Агентов (Work Packages)

Агенты, выполняющие модернизацию, обязаны придерживаться следующей последовательности этапов. Каждый этап должен компилироваться (`npm run typecheck`) и проходить тесты (`npm run check:readonly`).

### Рабочий Пакет 1 (WP-1): Офлайн-компилятор матрицы (`scripts/compile_markov_matrix.ts`)
**Владелец:** Скриптовый слой (`scripts/`).
**Задача:** Создать детерминированный генератор статических данных для браузера.
1. Написать `scripts/compile_markov_matrix.ts`, который объединяет результаты `discover_categories.ts` и `discover_patterns.ts`.
2. Применить строгий санитизационный фильтр (`categoryMap` / `blacklist`) ко всем извлеченным токенам.
3. Вычислить N-грамные переходы (униграммы, биграммы, триграммы) и привязать к каждому атому PCA-оси (`pcaDanger`, `pcaWealth`, `pcaNeed`).
4. Записать готовую, оптимизированную структуру в `src/data/markov_compiled_matrix.ts` (или экспортировать внутри `src/data/markov_text.ts`), чтобы в `dist/` она попадала как статическая типизированная константа без файловых операций.

### Рабочий Пакет 2 (WP-2): Расширение интентов и контекста (`src/systems/speech_router.ts` & `src/data/markov_text.ts`)
**Владелец:** Слой типов и данных (`src/systems/speech_router.ts`, `src/data/markov_text.ts`).
**Задача:** Расширить покрытие систем без нарушения обратной совместимости.
1. Добавить новые интенты в `MarkovIntent`: `document_flavor` (обрывки записок, журналы дежурных, инструкции ГО) и `lore_note`.
2. Добавить для них Fallback-строки в `MARKOV_INTENT_FALLBACKS`.
3. Обогатить `MarkovTextContext` явными числовыми метриками (если их нет в конкретном запросе, роутер автоматически выводит их из существующих `dangerBand`, `wealthBand`, `needBand`).

### Рабочий Пакет 3 (WP-3): Модернизация движка в рантайме (`src/systems/markov_text.ts`)
**Владелец:** Слой выполнения (`src/systems/markov_text.ts`).
**Задача:** Внедрить непрерывный скоринг и поиск по синтаксическим скелетам.
1. Подключить сгенерированные в WP-1 синтаксические скелеты (`<СУБЪЕКТ> -> <УГРОЗА>`, `<МЕСТО> -> <ЦЕННЫЙ_ПРЕДМЕТ>`) к системе рендеринга слотов (`generateSlot`).
2. Заменить или дополнить функцию `atomContextScore()` расчетом дистанции по осям PCA (`pcaDanger`, `pcaWealth`, `pcaNeed`).
3. Обеспечить сохранение жестких ограничений по длине (`maxChars`), детерминированности по `SeedRng` и нулевой аллокации в горячих циклах.

### Рабочий Пакет 4 (WP-4): Полная верификация и регрессионное тестирование (`tests/`)
**Владелец:** Слой валидации (`tests/unit/`).
**Задача:** Гарантировать абсолютную стабильность и каноничность.
1. Добавить unit-тесты в `tests/unit/markov_text.test.ts`, проверяющие генерацию новых интентов (`document_flavor`, `lore_note`).
2. Добавить автоматический тест-гарант на отсутствие запрещенных слов (`ЧАЭС`, `Сидорович`, `Контролер`, `Снорк`) в 1000 случайных прогонах генератора по всем интентам.
3. Убедиться, что `npm run check:readonly` (1800+ тестов, статический аудит контента) проходит с нулевым числом ошибок.

## 4. Контрольный Чеклист Агента перед Коммитом

- [ ] Код не использует `node:fs`, `path` или другие Node-specific модули внутри `src/`.
- [ ] Все случайные числа получаются только через `rng()` / `SeedRng` из `src/core/rand.ts` (запрет `Math.random()`).
- [ ] Ни одно слово из чужих франшиз не просачивается в `SpeechRouterResult.text`.
- [ ] `npm run typecheck` завершается с кодом 0.
- [ ] `npm run check:readonly` проходит без сбоев.
- [ ] `README.md` и `markov.md` остаются в актуальном состоянии относительно фактической реализации.
