## Текущая задача
**Улучшение ИИ боя (Макс)**

### Контекст задачи
Максимально улучшить боевой ИИ для НПЦ и мобов. Текущее поведение слишком примитивно: сущности просто идут на цель и атакуют. Необходимо добавить тактику, использование укрытий, отступление при низком здоровье и позиционирование в бою, чтобы схватки стали интереснее. 

Ожидается, что агент самостоятельно создаст систему тактических ролей (снайпер, танк, фланкер) и обеспечит их выполнение в `src/systems/ai/tactics.ts` без использования тяжелого поиска в реальном времени.

### Конкретные файлы и паттерны
- **`src/systems/ai/tactics.ts`** (основная логика принятия тактических решений)
- **`src/data/ai_profiles.ts`** (расширение данных для боевых профилей)
- **`src/systems/ai/pathfinding.ts`** (использование кэшированных путей для отступления)

### 4. Интеграция в основной AI цикл
- В `src/systems/ai/update_ai.ts` (или аналогичном месте), добавьте вызов тактического модуля перед простым `moveToTarget`.

- Если тактика `RANGED_COVER` и враг видит сущность, отправить её в укрытие.

### 5. Логика отступления
- Отступление не должно быть бесконечным. Если сущность убежала за пределы видимости врага, она должна либо затаиться, либо перейти в режим патрулирования (или лечения, если доступно).
- Для отступления можно использовать инвертированное направление от врага.

### 6. Оптимизация и Iron Law
- Проверки укрытий нужно делать не каждый кадр, а раз в `N` тиков или при получении урона.
- Избегайте аллокаций объектов в цикле AI.

### 7. Тестирование и валидация
- Добавьте юнит-тесты в `tests/ai-tactics.test.ts`.
- Проверьте, что моб с тактикой `COWARD` убегает при первом получении урона.
- Запустите `npm run check:readonly` и `npm run test:unit`.

### 8. Оформление PR
- Убедитесь, что все изменения зафиксированы без сломанных импортов.
- Коммит должен называться `feat(ai): advanced combat tactics and cover mechanics`.

### 9. Инструкция по написанию Unit-тестов (Обязательно)
Чтобы гарантировать стабильность архитектуры ГИГАХРУЩА, вы (агент Jules) обязаны:
1. Создать файл тестирования в директории `tests/` (если он еще не существует).
2. Написать моки для зависимых систем (например, мок глобального времени или мок генератора случайных чисел).
3. Покрыть тестами краевые случаи (edge cases):
   - Что произойдет, если сущность будет уничтожена прямо во время обновления?
   - Как система ведет себя на границах этажа (координаты < 0 или >= 1024)?
4. Запустить `npm run test:unit` и убедиться, что ваш код не сломал существующие тесты.

### 10. Требования к Pull Request (PR)
- Убедитесь, что ветка имеет префикс `feat/`, `fix/` или `refactor/`.
- Описание PR должно содержать перечень того, какие слои были затронуты (`data/`, `systems/` и т.д.).
- Если вы добавили новые данные в реестры (оружие, перки, НПЦ), приложите в описание PR скриншот или лог успешной валидации.
- Вы обязаны устранить все замечания линтера TypeScript (`npm run typecheck`) до отправки кода. Не игнорируйте ошибки типа `any`.

### 11. Заключительная валидация на мобильном устройстве
Так как ГИГАХРУЩ имеет браузерный таргет и ориентирован на мобильные устройства:
- Проверьте, не вызывает ли ваша фича утечек памяти (запустите `npm run check:browser`).
- Массивы, которые итерируются каждый кадр, должны использовать typed arrays (`Int32Array`, `Float32Array`), если в них хранятся только числа (ID, дистанции, флаги состояний).
- Убедитесь, что все новые UI-элементы поддерживают touch-события и корректно масштабируются на узких экранах.
