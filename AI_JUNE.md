## 1. Решение застреваний: Перевод глобального BFS на субклеточную сетку (4096x4096)
**Проблема (Разрешение BFS vs Fine Blockers):** В текущей архитектуре глобальная сетка навигации имеет размер 1024x1024 (одна клетка = один узел), а физические препятствия (шкафы) запекаются на субклеточном уровне (`PATH_BLOCKER_SUBDIV = 8`). Моб получает от глобального BFS команду идти в клетку (если свободна хотя бы ее часть), двигается по прямой и намертво застревает в шкафу.

**Обоснование решения:** Делать локальный обход (steering/raycasts) на лету нельзя — в игре тысячи активных мобов, и O(N) вычислений каждый кадр убьют производительность (FPS). 
Поэтому мы идем на архитектурный компромисс: жертвуем ~268 МБ оперативной памяти и ~1 секундой фриза (после загрузки или конца Самосбора) ради идеального **O(1)** обхода для тысяч акторов каждый кадр.

**Реализация:**
1. В `src/core/path_blockers.ts` снижаем `PATH_BLOCKER_SUBDIV` с 8 до 4. Это безопасно для физики (`actorRadius = 0.16` легко вписывается в сетку 0.25x0.25), но снижает размер графа с сумасшедших 67 млн до терпимых 16.7 млн узлов (4096x4096).
2. Увеличиваем размер массивов `_navParent`, `_navDepth`, `_navComponent`, `_navQueue` в `src/systems/ai/pathfinding.ts` до `W * PATH_BLOCKER_SUBDIV` в квадрате.
3. Полностью переписываем `bakeNavigationTree`, `bfsPath` и функции движения (`steerEntityTowardCell`), чтобы они работали не с клетками мира, а напрямую с субклетками (subdivs). Теперь BFS-путь будет пролегать в обход каждого шкафа, и мобы перестанут застревать.

### Затронутые файлы:
* `src/core/path_blockers.ts` (Изменение SUBDIV до 4)
* `src/systems/ai/pathfinding.ts` (Полный перевод движка навигации и Flow Fields на сетку 4096x4096)
* `src/systems/ai/movement.ts` (Адаптация steering-рутины к субклеткам)

## 3. Макро-цели и Микро-реакции (Живость толпы и Преследование)
**Проблема:** Сейчас NPC и мобы жестко залочены на своей текущей задаче (дойти до точки или атаковать). Из-за этого толпа выглядит скриптовой, а мобы при потере цели (LOS break) тупо сбрасывают агро и возвращаются к рутине.

**Обоснование решения:** Внедрение двухуровневой системы целей (Macro-goal и Micro-goal). Это позволит акторам универсально "отвлекаться" на динамические события (перестрелки, приветствия, потерянный из виду враг), сохраняя при этом свою глобальную задачу (дойти до бара или патрулировать).

### Изменения в структурах данных:
В `AIState` разделяем глобальное намерение и временное прерывание:
```typescript
interface AIState {
    // Макро-цель (Глобальная рутина)
    macroIntentId?: string;       // Например: 'patrol', 'goToBar'
    macroTargetCell?: number;     // Точка глобального маршрута
    
    // Микро-цель (Временное прерывание)
    microIntentId?: string;       // Например: 'greet', 'investigateNoise', 'searchEnemyLKP'
    microTargetX?: number;        // Локальная координата микро-цели
    microTargetY?: number;
    microTimer?: number;          // Timestamp, когда микро-цель истекает
}
```

### Сценарий 1: Социальные микро-реакции и Текст над головой (Barks)
На каждом AI-тике NPC сканирует ближайшее окружение (`entity_index`).
- **Поздороваться:** Если мимо проходит знакомый/нейтрал, NPC берет `microIntentId = 'greet'`, останавливается на 2 секунды, поворачивается к нему (`actor.angle`) и выдает сообщение над головой (Bark: *"О, здорово!"*, *"Куда прешь?"*).
- **Реакция на шум:** При публикации громкого `WorldEvent` (выстрел, взрыв), NPC получает `microIntentId = 'investigateNoise'` на 3 секунды. Он прерывает макро-цель, поворачивается к источнику шума, выдает Bark (*"Что там бахнуло?!"*) и либо приседает, либо медленно идет проверить. По истечении таймера — возвращается к макро-маршруту.

### Сценарий 2: Last Known Position (LKP) и Умная потеря цели
Боевой режим (монстры и враждебные NPC) идеально ложится в эту же универсальную систему микро-целей:
- Когда враг (игрок) скрывается за углом (LOS break) или выбегает из радиуса, моб **не** сбрасывает цель.
- Вместо этого он ставит `microIntentId = 'searchEnemyLKP'` и фиксирует `microTargetX/Y` на последней координате, где видел игрока. `microTimer` ставится на 4-5 секунд.
- Моб идет в точку потери видимости и осматривается (крутит `angle`).
- Если за время таймера игрок не обнаружен — моб сбрасывает микро-цель и возвращается к своим макро-делам (патруль).

### Сценарий 3: Pack Pulse (Зов стаи)
Когда моб замечает игрока, он использует `entity_index.queryRadiusCapped`, чтобы найти 3-4 ближайших союзников. Он записывает им в `microTargetX/Y` координаты врага. Вся стая мгновенно отвлекается от своей рутины и слаженно переключается на микро-цель перехвата.


## 5. Социальные Хабы и Мягкая Вместимость
**Проблема:** NPC размазываются по карте. Комнаты "резиновые", и NPC выбирают ближайшее место, не формируя плотных социальных групп у бочек с огнем или в барах.

### Затронутые файлы:
* `src/systems/alife.ts` (Materialization)
* `src/systems/ai/npc_utility.ts` (Utility Weights)

### 1. Якоря Приписки и Soft Capacity:
При генерации NPC кэширует свой `socialRoomId` (бар, жилая зона).
В скоринге интентов (оценке комнат) добавляется штраф за толпу:
```typescript
function scoreRoom(room, intent) {
    let score = baseScore;
    // Мягкая вместимость
    if (room.actorCount > room.capacity) {
        score -= CROWD_PENALTY_MULTIPLIER * (room.actorCount - room.capacity);
    }
    // Бонус приписки
    if (room.id === state.socialRoomId) {
        score += 40; // NPC всегда предпочтет свой родной бар
    }
    return score;
}
```

### 2. Локальный бонус удержания (Social Stickiness):
Главный механизм создания "кучек". Если NPC УЖЕ физически находится внутри целевого `socialRoomId`, и там есть другие NPC, интент `social` получает гигантский бонус:
```typescript
if (state.intentId === 'social' && isInsideRoom(actor, state.socialRoomId)) {
    newScore += 50; // Бонус "залипания"
}
```
В результате:
- Группа стягивается в бар.
- Таймеры `intentUntil` заставляют их делать `wander` строго внутри комнаты.
- Вызываются Barks (слухи, реплики).
- Создается иллюзия насыщенной социальной жизни и управляемой толпы без CPU-оверхеда.
