## Текущая задача
**Маленькая девочка и ликвидатор**

### Контекст задачи
Вдохновлено "Big Daddy и Little Sister" из BioShock. На каждом процедурном этаже (или с определенным шансом) генерируется уникальный дуэт: беззащитный НПЦ (Маленькая Девочка / Сборщица), собирающая редкий ресурс (например, ПСИ-пыль), и её опекун — Ликвидатор (смертоносный моб/НПЦ высокого уровня). Игрок может:
- Проигнорировать их.
- Убить Ликвидатора и забрать ресурс (дает огромный буст, но портит карму/сюжет).
- Спасти/Помочь девочке квестом (меньшая награда сразу, но перманентный бонус к статам).

### Конкретные файлы и паттерны
- **`src/data/npc_plot_packages.ts`** (определение дуэта)
- **`src/gen/living/zone_content.ts`** (генератор для спавна дуэта на этаже)
- **`src/systems/ai_tactics.ts`** (поведение "Опекун")

## Подробный Roadmap (Jules)
Этот план описывает создание дуэта, их ИИ и интеграцию в генерацию.

### 1. Определение сущностей (TypeScript Stubs)
- Добавьте в реестр НПЦ/Моб данные для Сборщицы и Ликвидатора.

```typescript

export const HARVESTER_DUO_PACKAGE = {
    harvesterDefId: 'npc_harvester_girl',

    resourceRewardId: 'item_psi_dust'

```

### 2. Специальный ИИ (Опекун)
- Ликвидатор пассивен, пока к Сборщице не подошли слишком близко или не атаковали её.
- Добавьте новое состояние в AI: `GUARDIAN`.

```typescript
// src/systems/ai_tactics.ts
export function updateGuardianAI(guardian: Entity, ward: Entity, target: Entity | null): void {
    // 1. Следовать за ward (Сборщицей), держа дистанцию ~2-3 клетки.
    // 2. Если target атакует ward или guardian — агрессия (MELEE_RUSH или RANGED).
    // 3. Если ward мертва — впасть в ярость (+статы, атаковать всех вокруг).
}
```

### 3. Логика Сборщицы (Harvester)
- Сборщица бродит между трупами мутантов или аномалиями и "собирает" ресурс.
- При смерти опекуна она убегает (flee) или плачет (открывается диалог).

### 4. Интеграция в генератор этажей
- Откройте `src/gen/population_placement.ts` или `zone_content.ts`.
- Добавьте шанс (например, 20% на нечетных этажах) заспавнить `HARVESTER_DUO_PACKAGE`.
- Свяжите их: запишите `wardId` в `guardian`, и `guardianId` в `ward`.

### 5. Взаимодействие и награды
- Реализуйте в `src/systems/interactions.ts` особый диалог при живом и при мертвом опекуне.
- Использование `item_psi_dust` дает перманентный бонус (используя функции из `rpg.ts`).
- Отправьте событие `publishEvent({ type: 'HARVESTER_KILLED' })` для влияния на карму.

### 6. Оптимизация и краевые случаи
- Что если их разделит стена? Ликвидатор должен уметь ломать гермодвери или использовать особый телепорт/рывок к подопечной.
- Избегайте постоянных вычислений пути: Ликвидатор просто двигается к координатам Сборщицы.

### 7. Тестирование
- Запустите `npm run test:generation` и убедитесь, что дуэт успешно размещается в комнатах.
- Напишите юнит-тест, где моб атакует Сборщицу, а Ликвидатор немедленно переключает агро на этого моба.
- Выполните `npm run check:readonly`.

### 8. Создание PR
- Убедитесь, что нет хардкода: имена, статы и айди лежат в `data/`.
- Коммит: `feat(content): Harvester and Liquidator guardian duo`.

### 9. Инструкция по написанию Unit-тестов (Обязательно)
Чтобы гарантировать стабильность архитектуры ГИГАХРУЩА, вы (агент Jules) обязаны:
1. Создать файл тестирования в директории `tests/` (если он еще не существует).
2. Написать моки для зависимых систем (например, мок глобального времени или мок генератора случайных чисел).
3. Покрыть тестами краевые случаи (edge cases):
   - Что произойдет, если сущность будет уничтожена прямо во время обновления?
   - Как система ведет себя на границах этажа (координаты < 0 или >= 1024)?
4. Запустить `npm run test:unit` и убедиться, что ваш код не сломал существующие тесты.

### 10. Требования к Pull Request (PR)
- Убедитесь, что ветка имеет префикс `feat/`, `fix/` или `refactor/`.
- Описание PR должно содержать перечень того, какие слои были затронуты (`data/`, `systems/` и т.д.).
- Если вы добавили новые данные в реестры (оружие, перки, НПЦ), приложите в описание PR скриншот или лог успешной валидации.
- Вы обязаны устранить все замечания линтера TypeScript (`npm run typecheck`) до отправки кода. Не игнорируйте ошибки типа `any`.

### 11. Заключительная валидация на мобильном устройстве
Так как ГИГАХРУЩ имеет браузерный таргет и ориентирован на мобильные устройства:
- Проверьте, не вызывает ли ваша фича утечек памяти (запустите `npm run check:browser`).
- Массивы, которые итерируются каждый кадр, должны использовать typed arrays (`Int32Array`, `Float32Array`), если в них хранятся только числа (ID, дистанции, флаги состояний).
- Убедитесь, что все новые UI-элементы поддерживают touch-события и корректно масштабируются на узких экранах.
