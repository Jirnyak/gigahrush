## Подробный Workflow реализации

### Шаг 1. Профилирование инициализации
1. Используйте Chrome DevTools (Performance tab) или встроенный таймер (`performance.now()`) в `main.ts` вокруг ключевых блоков (генерация, загрузка сохранений, WebGL).
2. Выявите самые медленные этапы (например, парсинг большого JSON сохранения, компиляция огромного атласа текстур процедурным генератором).

### Шаг 2. Оптимизация `JSON.parse` и сохранений
1. Если парсинг сохранения в `floor_memory.ts` занимает много времени, рассмотрите возможность ленивой загрузки (загружать только текущий этаж, а не все сохраненные этажи сразу).
2. Если в `localStorage` хранятся тяжелые объекты, используйте типизированные массивы (Base64 -> Uint8Array), которые десериализуются мгновенно по сравнению с огромными массивами чисел в JSON.

```typescript
// Stub: src/systems/save_runtime.ts (Optimization)
export function parseCompactFloorData(base64str: string): Uint8Array {
    // Avoids massive JSON parsing for raw cell data
    return decodeBase64ToBuffer(base64str);
}
```

### Шаг 3. Оптимизация процедурной генерации текстур
1. Если текстуры или спрайты (в `src/render/sprites.ts`) генерируются на лету через Canvas API попиксельно, это может блокировать Main Thread.
2. Отложите генерацию текстур, которые не нужны прямо сейчас (например, монстры с других этажей), либо используйте `ImageBitmap` и `OffscreenCanvas` асинхронно, если доступно.

```typescript
// Stub: src/render/textures.ts
export async function initTexturesAsync() {
    // Generate base textures first (walls, floors)
    await generateBaseTextures();
    
    // Defer advanced/rare textures to not block initial render
    setTimeout(() => {
        generateDeferredTextures();
    }, 100);
}
```

### Шаг 4. Оптимизация навигационного кэша (A*)
1. На этапе старта генерируется навигационный кэш этажа. Если он строится для всей карты 1024x1024, это дорого.
2. Проверьте, можно ли строить навигационный кэш только для "раскопанных" (проходимых) секторов, игнорируя целые массивы сплошных стен (Iron Law).

### Шаг 5. Показ заглушки (Loading Screen)
1. Если блокировка Main Thread неизбежна, убедитесь, что в `index.html` есть легковесный CSS-лоадер, который появляется ДО загрузки тяжелого `main.ts` бандла.
2. Разбейте инициализацию на чанки через `requestAnimationFrame`, чтобы браузер мог отрисовать лоадер.

```typescript
// Stub: src/main.ts
async function bootstrap() {
    ui.showLoader('Loading save...');
    await yieldToBrowser();
    loadSave();
    
    ui.showLoader('Generating world...');
    await yieldToBrowser();
    generateFloor();
    // ...
}
function yieldToBrowser() {
    return new Promise(r => setTimeout(r, 0));
}
```

### Шаг 6. Написание тестов производительности
1. Создайте `tests/benchmarks/load_time.ts`.
2. Замерьте время генерации этажа. Оно должно быть < 500ms для комфортной игры.

1. Соберите игру (`npm run build`).
2. Проверьте размер бандла.
3. Откройте в инкогнито и замерьте время от клика до первого кадра рендера. Оно должно заметно уменьшиться.
